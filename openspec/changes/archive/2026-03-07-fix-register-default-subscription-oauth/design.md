## 上下文

当前“注册默认赠送订阅”能力已经存在，但还有两个边界没有收好：

- 管理端保存默认套餐时，只会提交非空套餐值，且后端会把 `<= 0` 的套餐标识直接视为无效，导致管理员无法把配置恢复为空。
- GitHub OAuth 新用户理论上应复用注册后默认套餐发放链路，但实际使用中没有稳定拿到默认套餐，需要补一条贴近真实控制器回调的回归保障，并在必要时修正链路。

这次变更是对既有能力的补洞，而不是重做整套默认订阅体系。

## 目标/非目标

**目标：**
- 支持管理员在关闭该能力时清空默认套餐配置。
- 保持启用状态下的严格校验，不允许“开启功能但没有有效套餐”。
- 让 GitHub OAuth 首次注册与普通注册共享一致的默认套餐发放结果。
- 通过定向测试锁定上述行为，防止回归。

**非目标：**
- 不扩展为按渠道、邀请关系或用户组发放不同套餐。
- 不改变现有订阅支付、退款或生命周期规则。
- 不为历史已注册用户自动补发默认套餐。

## 决策

### 决策1: 将“空套餐”视为一种合法的禁用态配置

前端需要允许管理员显式提交空值，后端则需要把“关闭功能 + 空套餐”识别为合法配置；但“开启功能 + 空套餐”仍然必须被拒绝。

这样可以让配置语义完整：管理员既能设置默认套餐，也能明确撤销该设置，而不会破坏现有启用校验。

### 决策2: 继续复用现有注册后发放入口，但用 GitHub OAuth 回归测试锁定行为

默认套餐发放已经有统一的注册后处理入口，因此优先复用现有链路，而不是再为 GitHub 单独创建一套赠送逻辑。

实现上会先核对 GitHub OAuth 新用户创建后的收尾路径是否真正触发默认套餐发放；如存在分支遗漏，则修正到统一入口。随后补充控制器级测试，直接覆盖 GitHub OAuth 首次注册场景，确保行为与普通注册一致。
