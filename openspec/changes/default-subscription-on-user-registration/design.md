## 上下文

当前系统已经具备订阅套餐、支付回调入账、用户订阅生命周期维护和配置中心热更新能力，但“新用户注册后自动获得默认订阅套餐”这条链路尚不存在。该变更会同时触达注册流程、配置中心、订阅领域和管理端配置入口，属于跨模块行为变更。

从现有能力边界看，支付订单完结链路适用于付费购买，配置中心适合承载运行时可调整的开关和目标套餐，订阅领域负责真正创建或续期用户订阅。默认赠送订阅应复用订阅领域能力，但不应伪装成支付成功事件，也不能因为赠送失败而阻断用户注册主流程。

相关利益方包括：
- 运营或管理员：希望将某个已配置套餐作为注册福利，并可随时启用、停用或替换。
- 新注册用户：希望在注册后立即获得明确、可见的订阅权益。
- 后端维护者：需要保证幂等、可观测、跨数据库兼容，并避免把支付语义污染到注册流程。

## 目标 / 非目标

**目标：**
- 提供一个默认关闭的系统配置，用于指定“注册即赠送”的订阅套餐。
- 在新用户创建成功之后触发默认订阅发放，并保证同一用户最多领取一次。
- 复用现有订阅领域对象创建用户订阅，而不是绕行支付订单回调链路。
- 当配置无效或发放失败时，保留注册成功结果，并记录可追踪的错误信息。
- 让管理端能够安全配置目标套餐，并在运行时生效。

**非目标：**
- 不实现按邀请码、用户组、来源渠道等复杂条件发放规则。
- 不把该能力扩展为通用营销活动引擎。
- 不改变现有订阅支付、退款和预扣费规则。
- 不为历史存量用户做自动补发。

## 决策

### 1. 用配置中心承载默认发放开关与套餐标识

决策：新增运行时配置，至少包含是否启用默认发放以及默认套餐标识。管理端通过现有配置读写接口维护该值，后端在保存时做基础校验。

原因：
- 该能力属于运营策略，适合放进现有 Option 和 ConfigManager 体系，便于热更新和统一管理。
- 不需要新增独立表结构即可满足第一阶段需求，降低迁移成本。

备选方案：
- 在订阅套餐表上增加“默认注册赠送”字段。未采用，因为会把系统级策略耦合进单个套餐实体，也会增加多数据库迁移复杂度。
- 在注册接口请求参数里传入套餐。未采用，因为这会暴露不必要的客户端控制面并带来滥用风险。

### 2. 在用户注册成功后同步触发赠送逻辑，但失败不回滚注册

决策：把默认订阅发放放在用户记录已创建成功之后执行，作为注册流程的后置业务步骤；若赠送失败，仅记录日志和返回成功注册结果，不删除用户、不回滚注册。

原因：
- 用户注册是主事务，默认赠送只是附加福利，不能让无关的套餐配置问题阻塞核心注册转化。
- 同步触发可以让用户在首次进入系统时立即看到订阅权益，体验更直接。

备选方案：
- 异步队列发放。未采用作为第一阶段方案，因为实现更复杂，还需要补偿、任务可视化和延迟一致性处理。
- 赠送失败则整体回滚注册。未采用，因为会放大配置错误带来的业务损失。

### 3. 复用订阅领域能力直接创建用户订阅，不走支付订单完结链路

决策：在订阅领域新增或复用明确的“管理员或系统发放订阅”能力，直接创建或初始化用户订阅记录；不构造伪订单，不模拟支付回调。

原因：
- 支付订单链路的语义是用户完成购买后的入账，和注册赠送不是一回事。
- 复用订阅领域对象可以共享有效期、额度重置、状态维护等既有规则，同时避免污染支付数据。

备选方案：
- 伪造一笔零金额订单后调用订单完结逻辑。未采用，因为会混淆财务和运营语义，也可能影响统计报表。

### 4. 用显式幂等校验避免重复发放

决策：发放前检查用户是否已经因注册默认赠送获得过该权益，确保重试、并发或第三方登录重复回调不会造成重复发放。实现上优先通过用户订阅状态和赠送来源标记实现幂等。

原因：
- 注册流程可能存在请求重放、OAuth 多次回调或应用层重试。
- 默认赠送具备明确的一次性语义，需要防止同一用户被重复加额。

备选方案：
- 仅依赖调用路径保证一次执行。未采用，因为跨登录方式和异常重试时不可靠。

### 5. 无效配置按软失败处理，并增强可观测性

决策：若默认套餐不存在、被停用或不再满足发放条件，注册流程继续，但要记录结构化日志，方便管理员排查。管理端保存配置时也应尽可能提前校验套餐有效性。

原因：
- 该配置由运营维护，配置漂移是现实风险。
- 软失败能保护注册成功率，提前校验和日志能降低问题发现成本。

备选方案：
- 运行时发现无效配置时直接报错给用户。未采用，因为用户无法理解后台运营配置异常。

## 风险 / 权衡

- [重复发放] 注册重试、并发请求或多种登录方式可能导致同一用户领取多次 -> 通过订阅发放来源标记、用户级幂等校验和必要的唯一约束降低风险。
- [配置失效] 管理员删除或停用默认套餐后，注册赠送可能失效 -> 在保存配置时校验套餐状态，并在运行时软失败加日志告警。
- [领域边界不清] 若直接复用支付完结逻辑，后续报表和财务语义会被污染 -> 单独抽象系统发放订阅能力，和支付流程分开。
- [跨数据库实现差异] 若为幂等新增字段或约束，需要兼容 SQLite、MySQL、PostgreSQL -> 优先使用现有 GORM 模型抽象与兼容迁移模式，避免写数据库方言特有 SQL。
- [用户体验取舍] 同步发放让体验更好，但注册路径会多一次订阅处理 -> 第一阶段接受这点复杂度，后续若性能压力显著再考虑异步化。
