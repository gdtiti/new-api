## 新增需求

### 需求: 管理员可以配置注册默认订阅套餐
系统必须允许管理员配置“新用户注册默认赠送订阅”能力，并且必须支持启用状态与目标订阅套餐标识的读写和校验。

#### 场景: 保存有效的默认订阅配置
- **当** 管理员启用注册默认订阅功能并提交一个存在且可用的订阅套餐标识
- **那么** 系统必须保存该配置并使其可被后续注册流程读取

#### 场景: 提交无效的默认套餐配置
- **当** 管理员提交不存在、已停用或不可用于默认发放的订阅套餐标识
- **那么** 系统必须拒绝该配置并返回可识别的错误信息

#### 场景: 禁用功能时清空默认套餐
- **当** 管理员关闭注册默认订阅功能，并将目标订阅套餐清空为“无”
- **那么** 系统必须保存该空配置
- **并且** 后续读取配置时必须表现为“未配置默认套餐”

#### 场景: 启用功能时提交空套餐
- **当** 管理员启用注册默认订阅功能，但提交的目标订阅套餐为空
- **那么** 系统必须拒绝该配置
- **并且** 返回可识别的错误信息提示必须先选择有效套餐

### 需求: 新用户注册后自动获得默认订阅
当注册默认订阅功能已启用且配置有效时，系统必须在新用户创建成功后为该用户发放指定的默认订阅权益。

#### 场景: 普通注册成功后发放默认订阅
- **当** 新用户通过受支持的注册流程成功创建账号，且默认订阅配置处于启用状态
- **那么** 系统必须为该用户创建与配置套餐对应的订阅权益

#### 场景: 首次登录前即可看到已发放订阅
- **当** 用户完成注册且默认订阅发放成功
- **那么** 系统必须使该订阅在用户订阅查询视图中可见

#### 场景: GitHub OAuth 首次注册成功后发放默认订阅
- **当** 新用户通过 GitHub OAuth 首次成功注册账号，且默认订阅配置处于启用状态
- **那么** 系统必须为该用户创建与配置套餐对应的订阅权益
- **并且** 该订阅必须在用户订阅查询视图中可见

### 需求: 默认订阅发放必须具备幂等性
系统必须防止同一用户因注册重试、并发请求或重复回调而重复领取注册默认订阅。

#### 场景: 重复执行注册后置发放逻辑
- **当** 同一用户的默认订阅发放逻辑被重复触发
- **那么** 系统必须确保该用户最多只获得一次注册默认订阅权益

### 需求: 默认订阅发放失败不得阻断注册
如果默认订阅配置失效或发放过程失败，系统必须保留用户注册成功结果，并且不得因为赠送失败回滚已创建的用户账号。

#### 场景: 发放时发现默认套餐已失效
- **当** 用户注册成功后，系统在发放默认订阅时发现目标套餐已不存在或已停用
- **那么** 系统必须保留注册成功结果，并记录可追踪的失败信息

#### 场景: 订阅创建过程发生内部错误
- **当** 用户账号已成功创建，但默认订阅发放过程发生内部错误
- **那么** 系统必须返回或保留注册成功状态，并记录该错误以供排查
